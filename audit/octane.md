## Vulnerability findings

### 1. Permissionless opener-as-proposer in Moloch.openProposal with quorum configured enables immediate proposal cancellation (DoS) - High

> **Review: Valid observation, not a production blocker.** The griefing vector is real but mitigable in practice. `cancelProposal` requires zero tallies, so the attack only works if the attacker can front-run and cancel before any vote is cast. Key mitigations: (1) `castVote` auto-opens proposals atomically (line 352), so proposers can open+vote in one tx — or use `multicall` to open and vote together — preventing the cancel window entirely; (2) `proposalThreshold > 0` restricts who can open; (3) auto-futarchy (`autoFutarchyParam > 0`) sets a non-zero pool on open, blocking cancellation; (4) the attacker bears gas costs with no financial incentive beyond griefing; (5) proposer can always reissue with a new nonce. The secondary concern (earlier snapshot/TTL from front-run open) is low severity — the proposer's subsequent `castVote` proceeds normally on the already-opened proposal. Good v2 hardening candidate (e.g., require nonzero voting power even when threshold is 0, per Zellic #10).

The first caller of openProposal becomes proposerOf[id] and fixes createdAt and snapshotBlock. With quorum configured and no futarchy earmark, the opener can immediately cancel a zero-vote proposal, permanently tombstoning that id. Early opening also starts TTL and fixes an earlier snapshot, reducing the voting window and potentially excluding intended voters.

In Moloch.openProposal, the first call for a given id sets snapshotBlock[id] = block.number - 1, createdAt[id] = block.timestamp (if unset), supplySnapshot[id], and proposerOf[id] = msg.sender. Proposals with zero votes and a configured quorum (quorumAbsolute > 0 or quorumBps > 0) are considered Active. cancelProposal(id) is restricted to proposerOf[id] and only succeeds if state(id) == Active, no votes have been cast, the proposal is not queued, and the futarchy pool is zero. Therefore, if an attacker learns a proposal id before inclusion (e.g., via public mempool), they can front-run openProposal to become proposerOf and then immediately call cancelProposal to set executed[id] = true, permanently tombstoning the proposal. Separately, early opening starts the TTL countdown (proposalTTL) earlier than intended and fixes an earlier snapshot, which can reduce the effective voting window and disenfranchise voters who acquire voting power after that snapshot. While a nonzero proposalThreshold can limit who may open, any threshold-eligible actor can still exploit this. Enabling autoFutarchyParam with a nonzero earmark can block cancellation by making F.pool > 0 on open; otherwise, the attack remains feasible.

### 2. Premature NO futarchy resolution before TTL in Moloch.resolveFutarchyNo causes proposal freeze and futarchy pool/treasury drain - High

> **Review: Valid observation, not a production blocker.** The interaction is real: when futarchy is enabled, early NO voters who meet quorum can call `resolveFutarchyNo` to finalize the pot and freeze further voting (line 366: `F.enabled && F.resolved` blocks `castVote`), cutting short the TTL window and preventing a FOR comeback. This is a genuine tension between the futarchy resolution path and the voting lifecycle. In practice, this is mitigable: futarchy is most useful in broadly distributed DAOs where early quorum by one side is unlikely. In concentrated DAOs where a small coalition can reach quorum quickly, futarchy adds little value and should not be enabled. UI/documentation should guide DAOs accordingly. The voting freeze on resolution (line 366) exists to prevent receipt composition from changing after payouts are locked — removing it would create a different problem. Consider decoupling futarchy resolution from the voting freeze in v2, or requiring TTL expiry before NO-side resolution when futarchy is enabled.

resolveFutarchyNo allows early NO finalization based on a transient Defeated state (before TTL elapses), sets F.resolved=true and blocks castVote() for the proposal, freezing tallies. This enables per‑proposal governance DoS and lets attackers capture futarchy rewards; with minted loot/shares and ragequit enabled, this can drain treasury assets.

In Moloch.sol, resolveFutarchyNo(id) can be called whenever state(id) is Defeated or Expired without requiring the proposal’s TTL to elapse. Calling it sets F.resolved=true and F.winner=0 (NO) via _finalizeFutarchy, which computes payouts for the current NO receipt supply. After this, castVote() reverts for that proposal because it checks if (F.enabled && F.resolved), permanently freezing voting and preventing any tally change. state() can evaluate to Defeated before TTL ends if quorums (if configured) are met and forVotes <= againstVotes (or minYes not met). cancelVote() cannot help because it only works in Active state, and post-resolution no further voting can move the state. fundFutarchy is permissionless for ETH, and openProposal can auto-enable futarchy earmarks (autoFutarchyParam > 0) defaulting to minted loot, creating pools subject to early NO resolution. Early resolution lets an attacker lock in favorable NO receipt composition and claim the pool (ETH or minted tokens). If rewards are minted loot/shares and ragequit is enabled, the attacker can convert the minted receipts into treasury withdrawals, causing direct loss of principal and ensuring the proposal can never pass.

### 3. First-opener-controlled snapshot in Moloch.openProposal/state causes manipulable dynamic quorum and governance DoS/low-turnout passage - High

> **Review: Real concern, configuration-dependent. Variant of Zellic #10.** The snapshot-at-block-N-1 pattern itself is standard (OpenZeppelin Governor). However, the interaction with dynamic quorum (`quorumBps > 0`) creates a real griefing path: an attacker with access to a minting sale (`buyShares`) can inflate supply before block N, then ragequit after the snapshot, artificially raising the quorum denominator for that proposal. This makes the proposal harder to pass (or easier to defeat via low turnout). The attack requires: (1) active minting-mode sale, (2) `ragequittable` enabled, and (3) `quorumBps > 0` — all simultaneously. It is economically constrained (attacker must commit real ETH to buy shares, pays gas, and ragequit returns pro-rata assets not the original ETH amount if treasury composition changed). Not a production blocker, but a real configuration-dependent concern. **v2 hardening:** consider snapshotting total supply for quorum calculation at proposal creation time, or adding a cooldown between share purchase and ragequit.

The first caller to open a proposal fixes the snapshot at block.number - 1 and stores the total share supply used to compute the dynamic quorum threshold. Because opening is permissionless when proposalThreshold == 0 and users can change total supply across blocks via buyShares (sale mint) or ragequit (burn), an attacker can inflate or deflate the snapshot supply just before opening to raise or lower the quorum target. This can block legitimate proposals or enable passing harmful proposals with unusually low turnout.

In Moloch.openProposal(id), the contract sets snapshotBlock[id] = block.number - 1 and supplySnapshot[id] = shares.getPastTotalSupply(snap). The state(id) function uses this stored supply to enforce dynamic quorum: totalCast must be >= mulDiv(quorumBps, supplySnapshot[id], 10000) before a proposal can be Succeeded. When proposalThreshold == 0, openProposal is permissionless; castVote also auto-opens on first vote if unopened. Users can change total share supply across blocks via buyShares (if an active minting sale is configured) or via ragequit (if enabled), both of which write totalSupply checkpoints that are read by getPastTotalSupply. Therefore, an attacker can change totalSupply in block B, then open the proposal in block B+1 to anchor the dynamic quorum denominator to the manipulated supply. Inflating supply raises the quorum target and can DoS a proposal; deflating supply lowers the target and can allow a coalition to pass a harmful proposal with thin turnout. Majority (FOR > AGAINST) is still required, but the turnout gate is manipulable at the snapshot via block-level timing plus supply changes.

### 4. Public auto-futarchy enabling and reward token lock in openProposal causes minted reward extraction and treasury drain/governance capture - High

> **Review: Real concern, configuration-dependent.** The DAO configures `autoFutarchyParam` intentionally, but the interaction with minted rewards (`rewardToken = 0` → minted Loot via `address(1007)`) creates an aggregate exposure path: unlike shares/loot held by the DAO (which have balance checks at lines 328-334), minted rewards have no balance cap — `F.pool += amt` earmarks unbounded fresh minting. Combined with `proposalThreshold == 0`, an attacker can mass-open proposals, each auto-earmarking minted loot, and none can be cancelled (line 428: `F.pool != 0` blocks cancellation). After TTL expiry, NO resolution mints loot which can be ragequitted for treasury. The per-proposal `autoFutarchyCap` limits individual proposals but not aggregate exposure across many proposals. Mitigations: `proposalThreshold > 0` gates proposal creation behind real stake; TTL introduces time delay; governance can zero out `autoFutarchyParam` to stop the attack; using non-minted reward tokens (actual shares/loot held by DAO or ETH) has natural balance caps. Not a production blocker if `proposalThreshold > 0` or reward token is non-minted, but a real risk in permissionless + minted-reward configurations. **v2 hardening:** consider a global aggregate cap on auto-futarchy earmarks, or restrict minted rewards to require `proposalThreshold > 0`.

openProposal is public and, when autoFutarchyParam != 0, any caller can enable futarchy for a proposal id, lock the reward token for that id (mapping 0 to minted Loot), and earmark a pool without deposits if the reward is a minted asset. With TTL-enabled expiry, the attacker can finalize the NO outcome and cash out minted rewards; minted Loot can be ragequit to withdraw treasury assets, and minted Shares can be used to capture governance and drain funds.

When autoFutarchyParam is non-zero, openProposal (public, gated only by optional proposalThreshold) sets futarchy[id].enabled = true, deterministically sets futarchy[id].rewardToken (mapping rewardToken==0 to address(1007), minted Loot), and increments futarchy[id].pool by a computed amount without any per-id governance approval or external deposit if the reward is minted. Because proposal state transitions to Expired when proposalTTL is set and time elapses, resolveFutarchyNo can be called to finalize the NO side. An attacker who cast a minimal NO vote can then cashOutFutarchy to mint the earmarked rewards. If the reward is minted Loot and ragequittable is true, the attacker can immediately ragequit to withdraw a pro-rata share of all treasury assets. If the reward is minted Shares, the attacker mints governance power and can pass and execute draining proposals. The reward token is locked at first enablement for the id, and the 0->Loot remapping in this auto path contradicts the documented 0=ETH meaning, allowing unintended forced minted-Loot pots. Per-proposal caps can be bypassed by mass-opening many proposal ids; there is no global cap.

### 5. Order-dependent futarchy resolution in Moloch.sol (resolveFutarchyNo vs spendPermit) causes NO payouts despite successful execution - High

> **Review: Not a bug. Variant of Zellic #8.** Requires permit/proposal ID collision with specific futarchy timing. Very low likelihood, UI mitigates.

resolveFutarchyNo can finalize a futarchy pot to NO before a permit-based execution occurs; spendPermit then still executes the operation but cannot flip the futarchy outcome to YES, allowing NO payouts even though the operation ultimately succeeded.

In Moloch.sol, resolveFutarchyNo(id) finalizes futarchy (F.resolved = true, F.winner = 0) when state(id) is Defeated or Expired and executed[id] is false. spendPermit only verifies that a permit exists (isPermitReceipt[id]) and does not check the proposal state or F.resolved; it sets executed[id] = true, executes the call, and then calls _resolveFutarchyYes(id), which returns immediately if F.resolved is already true. As a result, a third party can finalize NO first when the proposal is Defeated or Expired, and a permit holder can later execute the same operation successfully; the futarchy pot remains NO-winner and pays NO receipt holders. Depending on rewardToken, payouts may mint new Shares/Loot (dilution) or transfer ETH/treasury tokens from the DAO.

### 6. Missing escrow for ETH futarchy funding in Moloch.sol causes ragequit to drain funds and block payouts - High

> **Review: Not a bug. Duplicate of Plainshift #2.** Ragequit gives pro-rata share of all DAO-held assets by design. Futarchy pools are incentive mechanisms subordinate to governance — not restrictive escrows. If excluded from ragequit, a hostile majority could shield treasury via futarchy funding.

ETH contributed to futarchy pools is not escrowed/locked. fundFutarchy(address(0), amount) credits F.pool but leaves the ETH in the DAO’s general balance. ragequit lets holders withdraw pro‑rata ETH without reducing F.pool. After resolution, cashOutFutarchy pays from live ETH using payoutPerUnit computed from F.pool; if ETH was drained, payouts revert. Depositors’ ETH can be diverted to ragequitters before winners can claim.

In Moloch.sol, when futarchy is funded with ETH (token == address(0)), fundFutarchy requires msg.value == amount and increments futarchy[id].pool, but it does not segregate or lock the received ETH; it remains in address(this).balance. ragequit permits ETH withdrawals (address(0) allowed) and computes a pro‑rata due from the current ETH balance, then pays via _payout(address(0), ...), reducing the DAO’s ETH without adjusting F.pool. After the proposal resolves, _finalizeFutarchy fixes payoutPerUnit = F.pool * 1e18 / finalWinningSupply. cashOutFutarchy burns winning receipts and calls _payout(address(0), claimer, payout); safeTransferETH reverts if the contract lacks sufficient ETH. Thus, ETH futarchy obligations (tracked by F.pool/payoutPerUnit) can exceed live reserves due to ragequit, causing payout failures. Additionally, ETH deposited by a funder can be partially or fully withdrawn by ragequitters before winners redeem, diverting principal. This issue is specific to ETH rewardToken configurations; minted token rewards (address(this)/address(1007)) mint on payout, and shares/loot pools cannot be drained by ragequit as those tokens are explicitly excluded there.

### 7. First-funder-controlled reward token selection in fundFutarchy (no preset/auto-futarchy) causes forced payout asset and funding path steering - Medium

> **Review: Not a bug.** Explicitly mitigated by governance setting `rewardToken` via `setFutarchyRewardToken`. Without a preset, first-funder choice is the intended fallback. DAO can always override via governance.

When no governance preset is set and auto-futarchy is off, the first successful fundFutarchy call locks the proposal’s reward token to the caller’s chosen asset (ETH, Shares ERC20, or Loot ERC20). This enables front-running to force the payout asset and require all subsequent funders to use that asset or fail.

In Moloch.sol, fundFutarchy sets FutarchyConfig.rewardToken on the first successful funding if FutarchyConfig.enabled is false and the global rewardToken preset is unset. It then marks FutarchyConfig.enabled = true. All subsequent fundings must use that exact token or revert. Pull semantics enforce: ETH requires msg.value == amount; ERC20 Shares/Loot use safeTransferFrom; minted sentinels (address(this) or address(1007)) are guarded to only be callable by the DAO and revert for EOAs (reverting rolls back state), so EOAs cannot choose minted assets. Thus, an external first funder can fix the reward token to ETH, Shares ERC20, or Loot ERC20, steering both subsequent funding and future payouts (cashOutFutarchy uses _payout with the locked-in token). Governance can mitigate by pre-setting the reward token (setFutarchyRewardToken) or enabling auto-futarchy (setAutoFutarchy) to pre-fix it on openProposal; absent these, the behavior enables front-running/first-mover asset selection.

### 8. Zero-address transfers without burn in Loot (and corner-case Shares) cause ragequit dilution and stranded funds - Medium

> **Review: Not a bug.** Transferring to `address(0)` is a user error. ERC-20 standard doesn't mandate `transfer(address(0))` as a burn. Badges seating logic blocks most Shares cases. Self-inflicted loss with minimal dilution impact.

Loot and Shares allow transfers to address(0) without reducing totalSupply. Moloch.ragequit computes payouts using shares.totalSupply() + loot.totalSupply() before burning the caller’s tokens, so any balances stranded at address(0) still count in the denominator, diluting payouts and leaving a proportional residue in the treasury that users cannot claim. For Shares, most zero-address transfers revert due to Badges seating logic, but a feasible corner case exists when seats are full and the zero-address balance does not beat the cutline.

In Loot and Shares, transfer/transferFrom call internal _moveTokens which updates balances and emits Transfer but does not treat to == address(0) as a burn or reduce totalSupply. In Moloch.ragequit, the denominator total is computed as shares.totalSupply() + loot.totalSupply() prior to the caller’s burns, and due = mulDiv(pool, amt, total) per asset. If tokens are sent to address(0), their balances remain counted in totalSupply but are unspendable (address(0) cannot ragequit). This reduces each exiting user’s due payout compared to a denominator that excluded blackholed balances and leaves a proportional residue in the treasury unclaimable by users. The DAO can later mitigate via onlyDAO burns or transfer locks, but that requires governance action and is not automatic. For Shares, Badges.onSharesChanged usually blocks transfers to address(0) because mintSeat(to) disallows to == address(0); however, when seats are full (256) and the zero address’s resulting balance is at or below the cutline (minBal), the code takes a no-op path, allowing the transfer and creating the same dilution effect.

### 9. Non-reserved balance-based auto-futarchy earmark in openProposal with local reward tokens causes overcommitment and payout denial/drain - Medium

> **Review: Not a bug. Variant of Zellic #23 / Plainshift #2.** Futarchy pools are earmarks, not escrow. Payouts bounded by actual balances at claim time. DAO governance controls pool creation and can negate via proposals or config changes.

When auto-futarchy is enabled and the reward token is configured to local Shares or Loot, openProposal computes each proposal’s futarchy pool from the DAO’s current token balance and adds it to F.pool without reserving or moving tokens. Multiple proposals can each earmark the same balance, enabling early claimants to drain the DAO’s local token treasury and causing later claims to revert. If Shares are used, this can also transfer governance power to early claimants.

In openProposal, if autoFutarchyParam > 0, the contract enables futarchy for the proposal and sets F.rewardToken. If the reward token is local (address(shares) or address(loot)), it computes an amount, clamps it to the DAO’s current balance of that token, and adds it to F.pool, but does not reserve or transfer any tokens. This allows the same DAO-held tokens to be overcommitted across many proposals opened while the balance is present. Later, when cashOutFutarchy is called, payouts for local tokens use safeTransfer from the DAO’s current balance; thus early claimants can drain the balance on one resolved proposal, and subsequent claims for other proposals revert due to insufficient balance. Transfers to the DAO are allowed even when transfersLocked is true, so an attacker can inflate earmarks by depositing local tokens before proposals open and then reclaim them by being first to claim. If rewardToken is minted tokens (address(1007) or address(this)), payouts mint on demand and this overcommit does not occur. Per-proposal caps limit only individual earmarks, not aggregate overcommit across many proposals. The No-resolution path requires proposalTTL > 0 to reach Expired (state() returns Expired when TTL passes); Yes-resolution requires passing quorums. The risk is configuration-dependent: it arises when autoFutarchyParam > 0 and rewardToken is set to local Shares or Loot while the DAO holds those tokens.

### 10. Missing decimals normalization in Moloch.buyShares pricing causes unusable or mispriced token sales - Medium

> **Review: Not a bug. UX/documentation concern.** `pricePerShare` is in raw base units — the DAO sets it knowing the token's decimals. UI should help DAOs set sane prices for different decimal tokens.

buyShares multiplies raw 18-decimal share amounts by payToken base units with no decimals normalization, preventing sane per-whole-share pricing for 6-decimal tokens (e.g., USDC) and forcing coarse granularity for ETH (<1 ETH per share cannot be expressed). This breaks intended token sales in common configurations.

In Moloch.buyShares, the cost is computed as cost = shareAmount * pricePerShare, where shareAmount is the raw amount of shares/loot (18 decimals) and pricePerShare is in payToken base units (wei for ETH). There is no normalization for differing token decimals. Shares and Loot both have 18 decimals. As a result, for 6-decimal tokens like USDC/USDT, the smallest nonzero slope implies a cost of 1e12 tokens for 1 full share (1e18 raw units), making sales economically impossible. For ETH/WETH (18 decimals), the smallest nonzero slope implies a minimum of 1 ETH per full share, so sub-1 ETH per-share pricing cannot be expressed; only fractional-share purchases can approximate desired economics. No code path normalizes payToken decimals, and maxPay only caps payment but does not solve the representational mismatch. This is a design-level pricing flaw that breaks an important non-core feature (token sales) for realistic configurations.

### 11. Snapshot/current-supply mix in auto-futarchy earmark basis in openProposal causes treasury drain via minted loot and ragequit - Medium

> **Review: Not a bug. Variant of Zellic #14.** Loot supply not snapshotted for futarchy basis is a design tradeoff. Futarchy pools are subordinate to governance. Practical exploitation requires loot sale + auto-futarchy + ragequit all enabled, with real economic cost. Consider snapshotting loot in v2.

The auto-futarchy earmark basis combines shares supply at the proposal’s snapshot block with current loot supply when the reward token is loot/1007. With minted rewards and ragequit enabled, an attacker can inflate loot supply just before opening a proposal to enlarge the earmarked pool, claim minted loot upon futarchy resolution, and ragequit to extract DAO treasury assets.

In openProposal, the DAO snapshots shares supply at block.number - 1 via getPastTotalSupply and stores it. If autoFutarchyParam > 0, it initializes/uses F.rewardToken (defaulting to address(1007) = minted loot if unset). It then computes basis = snapshot shares supply, and if the reward token is loot/1007, adds the current loot.totalSupply() (no checkpoints). The earmark amount amt is p% of this basis (or absolute if >10_000), and if the reward token is minted (1007 or address(this)), the code increments F.pool without any balance cap or asset transfer. Upon resolution, payoutPerUnit = pool / finalWinningSupply, and cashOutFutarchy mints the reward token to claimers. Ragequit pays out a pro-rata share of treasury based on (sharesToBurn + lootToBurn) over (shares.totalSupply + loot.totalSupply) measured pre-burn. This snapshot/current mix lets an attacker deterministically inflate loot supply right before opening to enlarge the earmark; the minted reward path allows unbounded minting of loot on cash-out; and ragequit converts that minted loot into real assets withdrawn from the treasury. The exploitability depends on DAO configuration (autoFutarchyParam, rewardToken choice, ragequittable, quorum/TTL), but under plausible settings it enables significant treasury loss.

### 12. Mixed snapshot/live supply basis in openProposal auto-futarchy earmark causes transaction-ordering-dependent mis-sizing of futarchy pool - Low

> **Review: Not a bug. Duplicate of #11 above.** Same underlying observation about loot supply not being snapshotted. The finding itself notes "does not by itself break invariants or directly steal funds."

openProposal snapshots share supply at block N-1 but reads loot totalSupply live when reward token is loot/1007, allowing attackers to change loot supply just before opening a proposal to under- or over-allocate the futarchy earmark.

In Moloch.openProposal, the share supply used for auto-futarchy earmarking is fixed via Shares.getPastTotalSupply at block.number-1, while for loot-based rewards (address(loot) or address(1007), which is the default when rewardToken == 0), the code adds the current Loot.totalSupply. Loot has no snapshots, so attackers can change loot.totalSupply immediately before openProposal (e.g., via ragequit burns to reduce, or via DAO-enabled loot mints) to influence the earmark amount. This creates a TOCTOU, transaction-ordering-dependent earmark that can result in smaller or larger futarchy pools than a consistent snapshot approach. Minted reward tokens are not clamped at earmark time (only local tokens are), though autoFutarchyCap limits magnitude. The issue does not by itself break invariants or directly steal funds; it primarily affects earmark sizing and incentives, with only conditional, economically constrained paths to treasury outflow.

### 13. Unbounded voting (proposalTTL==0) in Moloch.sol allows timelock-period vote flips causing execution denial - Low

> **Review: Not a bug.** The finding correctly notes this is "a consequence of the configuration choice." If the DAO sets `proposalTTL = 0`, open-ended voting is intentional. Liveness concern, not asset loss.

When proposalTTL is set to 0, voting never closes. Proposals can be queued as Succeeded, but additional AGAINST votes can be cast during the timelock window. Since state() only evaluates success/defeat at timelock maturity and executeByVotes re-checks state at execution-time, an attacker with unused snapshot voting power can flip a proposal to Defeated at maturity and block execution.

In Moloch.sol, castVote enforces expiry only if proposalTTL != 0; with proposalTTL == 0, voting is open indefinitely until executed[id] is set. state() returns Queued for queued proposals until queuedAt + timelockDelay, deferring any success/defeat evaluation. queue() requires the proposal to be Succeeded at call-time but does not freeze tallies or disable voting. executeByVotes always re-checks state at call-time, and after timelock maturity requires state(id) == Succeeded or it reverts. Therefore, with no TTL, an attacker holding unused snapshot voting power can cast late AGAINST votes during the timelock (or immediately before execution when no timelock), flipping the outcome to Defeated at the decisive moment and causing execution to fail. This yields a governance liveness/availability risk rather than asset loss, and is a consequence of the configuration choice (no voting deadline).

### 14. Auto-futarchy earmark in openProposal (autoFutarchyParam > 0) causes denial of cancellation - Low

> **Review: Not a bug.** Intentional — futarchy-funded proposals shouldn't be unilaterally cancelable because incentive funds are committed. The finding acknowledges this is "configuration-driven."

With autoFutarchyParam enabled, opening a proposal auto-enables futarchy and sets a non-zero futarchy pool (especially for minted reward tokens). cancelProposal then reverts if F.enabled && F.pool != 0, making cancellation of that proposal id impossible even with zero tallies.

When a proposal is first opened, openProposal records the snapshot and, if autoFutarchyParam > 0, enables futarchy for that id and computes an earmark amount (BPS of snapshot-based basis or absolute). If rewardToken is unset it defaults to a minted loot sentinel; minted reward tokens are not clamped to the DAO’s balance, so the computed amount typically results in F.pool > 0. cancelProposal requires zero tallies and Active state but additionally reverts if F.enabled && F.pool != 0, so any proposal opened under these settings cannot be canceled. No funds are moved for minted rewards until payout, but the non-zero pool still blocks cancellation. This is configuration-driven and can be avoided by disabling auto-futarchy, choosing a reward token that clamps to 0, or relying on TTL/republishing with a new nonce.

### 15. Non-snapshotted quorum/threshold parameters in proposal state evaluation cause retroactive outcome changes and execution blockage - Low

> **Review: Not a bug. Duplicate of Zellic #6.** Deliberate design choice. Standard pattern (early Compound Governor). Snapshotting params per-proposal adds significant storage cost. Parameter changes require governance approval from the same majority affected.

Proposal outcomes are recomputed against mutable, global quorum/threshold settings on every state() call. Because these parameters are not snapshotted per proposal, changing them mid-lifecycle (even after queuing) can flip a previously passing proposal back to Active/Defeated. With a nonzero TTL and nonzero timelock, this can strand the proposal id and block execution; futarchy payouts can also become misaligned with eventual execution.

The governance gates (quorumAbsolute, quorumBps, minYesVotesAbsolute) are global and updated in place. The state(id) function re-evaluates a proposal’s status using the current values each time it is called; there is no per-proposal snapshot of these parameters. When a proposal is queued, only the timelock countdown is recorded; after the delay elapses, state() again checks the current thresholds rather than treating the proposal as irrevocably passing. Meanwhile, castVote() enforces proposalTTL, so after TTL expires no new votes can be added to meet newly raised thresholds, although cancelVote() remains available when state() returns Active. Consequently, raising thresholds mid-timelock can block execution of a previously passing and queued proposal, requiring re-proposal. Additionally, futarchy resolution can be locked to NO during a temporary Defeated state and remain misaligned even if the proposal is later executed.

### 16. Missing executedcheck in Moloch.openProposal/fundFutarchy causes unresolvable futarchy deposits (user funds stuck) - Low

> **Review: Not a bug. Duplicate of Zellic #8.** Requires deliberately opening/funding a proposal for an already-executed permit intent. Very low likelihood, UI mitigates. Funds remain in DAO balance, accessible via governance.

openProposal and fundFutarchy do not check executed[id]. After an intent is executed via spendPermit without being opened, these functions can register and accept futarchy deposits on an already-executed proposal, creating an unresolvable pot that traps user funds; also causes a registry/state mismatch.

In Moloch.openProposal, the only early-exit is snapshotBlock[id] != 0; there is no check for executed[id]. If an id was executed via spendPermit before opening, executed[id] = true while snapshotBlock[id] == 0. ### 17. First-opener-based proposerOf authorization in cancelProposal causes id-level proposal DoS - Low

### 21. Global nonReentrant on executeByVotes in Moloch causes denial-of-execution for proposals invoking nonReentrant endpoints - Low

### 22. Double-floor rounding in Moloch.sol futarchy payout causes small unclaimable dust and underpayment to winners - Informational

> **Review: Not a bug. Informational.** The finding itself states "no theft path exists; this is a benign rounding artifact." Wei-level dust is standard in Solidity math.

Futarchy payouts floor twice: payoutPerUnit = floor(pool * 1e18 / finalWinningSupply) and per-claim payout = floor(amount * payoutPerUnit / 1e18). This guarantees a small unclaimable remainder (dust) when the division is not exact. With ETH/Shares/Loot (all 18 decimals), the dust is bounded to tiny wei amounts; leftovers remain in the DAO for ETH/ERC20 rewards or manifest as slight under-minting for mint-on-demand rewards.

On finalization (_finalizeFutarchy), payoutPerUnit is computed as ppu = mulDiv(pool, 1e18, finalWinningSupply), which floors. On claim (cashOutFutarchy), payout = mulDiv(amount, ppu, 1e18), which also floors. The two floors ensure the total of all claims is at most pool minus a small remainder δ = ceil(((pool * 1e18) % finalWinningSupply) / 1e18). With reward tokens restricted to ETH, Shares, or Loot (18 decimals), this remainder is always a tiny amount of wei-of-reward. Fragmenting claims can add at most one extra wei lost per additional claim. There is no residual reconciliation; leftover ETH/ERC20 remains in the DAO, and for mint-on-demand rewards the shortfall simply appears as under-minting relative to the earmarked pool. No theft path exists; this is a benign rounding artifact.

## Warning findings

### 1. Missing executed latch check in permit path (spendPermit) allows replayed execution causing duplicate payouts/mints - Medium

> **Review: False positive. Duplicate of Zellic #12.** Per-receipt execution is by design. Permit receipts are fungible ERC-6909 tokens — multiple receipts authorize multiple executions. The receipt balance burn is the replay guard. Re-minting receipts post-execution requires `onlyDAO` governance approval, which is an intentional action.

The spendPermit function executes intents without checking the executed[id] latch, enabling a permit holder to re-execute an already executed or cancelled intent if a permit receipt remains or is re-minted, leading to duplicate transfers or unintended minting.

In Moloch.sol, executeByVotes enforces one-shot semantics by reverting if executed[id] is true and then setting executed[id] = true before _execute, and cancelProposal tombstones an id by setting executed[id] = true. However, spendPermit only requires isPermitReceipt[id] and the ability to burn one receipt; it unconditionally sets executed[id] = true and calls _execute without checking whether executed[id] is already true, nor does it enforce proposal state or timelock. Additionally, setPermit (onlyDAO) can mint receipts even after an id has been executed or cancelled. As a result, if the DAO has ever minted a permit receipt for a given intent id and that receipt remains outstanding (or is later re-minted), the permit holder can call spendPermit to re-execute the same intent even after governance has executed or cancelled it, causing repeated payouts or incorrect additional minting.

### 2. Unbound target code for delegatecall proposals in Moloch.executeByVotes/_execute causes arbitrary DAO-context execution and fund/governance compromise - Medium

> **Review: Design tradeoff, not a bug.** Delegatecall (op==1) is an advanced governance tool that requires voter trust in the target. Code-hash binding would break legitimate upgradeable/proxy targets. The timelock provides a review window. DAOs using delegatecall proposals should exercise appropriate caution — this is inherent to the delegatecall model across all governance frameworks.

Proposal IDs do not bind the target’s code for delegatecall (op==1). A proposer can have voters approve benign behavior and then change or deploy the code at the same target address before execution, causing the DAO to delegatecall malicious logic that runs in the DAO’s storage context, enabling treasury drain and arbitrary minting of shares/loot.

In Moloch, proposal IDs are computed by _intentHashId(address(this), op, to, value, keccak256(data), nonce, config) and do not include any binding to the target contract’s code (e.g., extcodehash(to)). Execution via executeByVotes calls _execute; for op==1, it performs to.delegatecall(data) without allowlisting, code-hash checks, or extcodesize checks. Delegatecalled code runs in Moloch’s storage/context, so external calls made by that code observe msg.sender == Moloch and pass onlyDAO guards in Shares and Loot (e.g., mintFromMoloch). Because the proposal hash is not code-bound, a proposer can point to a target whose effective logic changes after voting but before execution—such as a beacon/registry-based delegatecall router or a CREATE2 counterfactual address—so the executed code differs from what voters reviewed. Timelock (if configured) only delays execution and increases the window for post-vote changes; it does not bind code.

### 3. EIP-1153 transient storage dependency in nonReentrant modifier causes DoS of critical functions on non-EIP-1153 networks - Medium

> **Review: Not a bug. Known deployment requirement.** The contract targets Cancun+ networks. EIP-1153 and EIP-3855 (PUSH0) are both required. Deployment to unsupported chains would fail at CREATE2, preventing any fund risk. Document as a deployment prerequisite.

The nonReentrant modifier in Moloch.sol uses EIP-1153 tload/tstore opcodes. On networks that have not enabled EIP-1153, calls to any function guarded by this modifier revert immediately with an invalid opcode, bricking governance execution, ragequit, futarchy claims, sales, and allowance withdrawals.

Moloch.sol implements a nonReentrant modifier that unconditionally executes EIP-1153 opcodes tload/tstore to manage a reentrancy guard. On chains where EIP-1153 is not enabled, these opcodes are unknown and cause an invalid-instruction revert at function entry. All flows protected by nonReentrant therefore fail before any logic runs. The affected functions include executeByVotes (governance execution), spendPermit (permit-based execution), cashOutFutarchy (claiming futarchy rewards), buyShares (token sale), ragequit (member exit/redemption), and spendAllowance (withdrawing allowances). Because onlyDAO methods require msg.sender == address(this), there is no alternative path to perform governance-controlled actions without successful execution by the DAO itself. Multicall/delegatecall cannot bypass onlyDAO or the nonReentrant guard. On Cancun/EIP-1153-enabled networks the design works as intended; the risk is a deployment-compatibility failure on unsupported networks, resulting in persistent DoS of critical functionality and potential fund freezing.

### 4. Pre-execution latch set before delegatecall in Moloch.executeByVotes allows callee to clear it, enabling repeated proposal execution and fund drains - Medium

> **Review: Design tradeoff.** Delegatecall inherently grants the callee full storage access — this is the same trust assumption as #2. A malicious delegatecall target can do anything to DAO storage, not just clear `executed`. Voters must trust the target code. This is intrinsic to delegatecall governance across all frameworks, not specific to Moloch.

In the delegatecall execution path (op==1), Moloch sets executed[id] = true before calling into attacker-controlled code and never re-asserts it after. The callee can clear executed[id] during delegatecall, enabling the same proposal to be executed again in later transactions. This breaks one-time execution semantics. Permits are not overspent because spendPermit also burns a unit per spend.

executeByVotes computes id and, after passing state/timelock checks, sets executed[id] = true, then calls _execute. When op == 1, _execute uses delegatecall, allowing the callee to write to Moloch’s storage. The callee can recompute id (using address(this), op, to, value, keccak256(data), nonce, and config), derive the executed mapping slot, and set it back to 0 during delegatecall. ### 5. Lack of safeguards between fixed-price sales and immediate ragequit in Moloch.sol causes treasury value extraction - Medium

> **Review: Design tradeoff, not a bug.** The DAO sets the sale price knowing ragequit is enabled. If `pricePerShare < NAV`, that's a governance decision (e.g., discounted entry for new members). ### 10. Missing executed-state check in fundFutarchy in Moloch futarchy funding allows post-execution/cancelled funding, causing unresolvable futarchy and stuck funds - Medium

> **Review: Not a bug. Duplicate of Zellic #8 / vuln #16/#20.** Same permit/futarchy ordering observation. Funds remain in DAO balance, accessible via governance. Very low likelihood — requires deliberately funding an already-executed intent. UI mitigates.

fundFutarchy lacks a check for executed[id]. ### 11. Per-asset floor division in Moloch.ragequit causes order-dependent underpayment to early quitters - Medium

> **Review: Not a bug. Informational.** Standard integer division behavior in Solidity. Bounded to <1 smallest unit per token per ragequit. Negligible for 18-decimal tokens. Universal across all pro-rata distribution contracts.

Ragequit computes each asset’s payout with integer floor division, which rounds down per token. The leftover remainder stays in the DAO until later ragequits, allowing late quitters to capture accumulated residuals. This creates an order-dependent fairness issue that can be material for low-decimal tokens.

In Moloch.ragequit, the amount paid per token is due = mulDiv(pool, sharesToBurn + lootToBurn, totalSharesAndLootBeforeBurn). mulDiv uses integer division, flooring each asset’s payout independently. Any per-asset remainder (pool * amt % total) is not reconciled and stays in the DAO’s balance for that token until future ragequits. The last (or late) quitter for that token receives whatever remains, leading to order-dependent payouts. This behavior is bounded by <1 smallest unit per token per ragequit; it is negligible for high-decimal tokens but can be material for 0‑decimal or very small integer pools. The code enforces sorted token lists and excludes DAO/shares/loot/sentinel addresses, but it does not reconcile rounding across assets or over time.

### 12. Config bump invalidates proposal/permit IDs without tombstoning in Moloch.sol causes futarchy payout DoS and unexecutable pre-bump actions - Low

> **Review: Not a bug. Duplicate of vuln #23.** Invalidation of pre-bump IDs is the intended purpose of `bumpConfig`. The DAO should resolve in-flight proposals before bumping. Document as operational guidance.

bumpConfig increments a global salt used in proposal/permit ID hashes; executeByVotes and spendPermit recompute IDs using the current salt, making pre-bump proposals unexecutable and pre-bump permits unspendable. Old proposals can still be opened, voted, and queued, which can indefinitely block futarchy resolution.

In Moloch.sol, _intentHashId(op,to,value,keccak256(data),nonce,config) includes the global config salt. bumpConfig() increments config but does not tombstone or block interactions with pre-bump IDs. executeByVotes and spendPermit always recompute IDs using the current config, so pre-bump proposals cannot be executed and pre-bump permits cannot be spent. queue(id) is permissionless; once a proposal is queued, state() disables TTL-based expiry. ### 13. uint96 checkpoint cast in Shares total-supply checkpoints causes minting-path DoS at 2^96 cap - Low

> **Review: Not a bug. Informational.** `uint96` supports up to ~79 billion tokens at 18 decimals. Standard practice (OpenZeppelin ERC20Votes uses uint224, Solady uses uint96). Reaching the cap requires unrealistic supply. Duplicate of #15 below.

Shares stores checkpointed totals/votes as uint96 and hard-casts on every mint/burn; once totalSupply would exceed 2^96−1, any further share mints revert, blocking all features that require minting shares.

In the in-file Shares contract, vote/total-supply checkpoints use struct Checkpoint { uint48 fromBlock; uint96 votes; }. _writeTotalSupplyCheckpoint() computes newVal = totalSupply and calls _writeCheckpoint, which writes votes = toUint96(newVal). toUint96 reverts if newVal ≥ 2^96. Every mint path (e.g., Shares.mintFromMoloch via _mint) updates totalSupply and then writes a checkpoint, so any mint that would push totalSupply above 2^96−1 reverts. This imposes a hard, implicit cap on total shares. Voting and burns remain functional because the overflowed state is never entered; the transaction attempting to exceed the cap reverts atomically. In Moloch, multiple features depend on minting shares: _payout(token == address(this)) mints shares, buyShares in minting mode mints shares, and cashOutFutarchy pays winners via _payout which can mint shares if the futarchy reward token is set to address(this). When totalSupply nears the cap, these mint-dependent paths revert and become unavailable.

### 16. PUSH0-based minimal proxy initcode in clone deployment on pre-Shanghai networks causes deployment/initialization DoS - Low

> **Review: Not a bug. Duplicate of #3.** Same deployment-compatibility observation. Contract targets modern EVM networks. The finding itself notes "likelihood is low given modern networks and the code's EIP-1153 dependency."

### 18. Push-only ETH refund to msg.sender with revert-on-failure in buyShares ETH overpayment path causes purchase revert for non-payable callers - Informational

### 22. Naive 256-bit mulDiv in ragequit payout calculation causes overflow revert and minor liveness/UX degradation - Informational

> **Review: Not a bug. Informational.** The finding itself notes "under realistic treasuries, reaching overflow is highly unlikely" and "minor liveness/UX degradation rather than loss of funds." Users can split ragequits as a workaround.

### 23. Zero-address seat minting check in Badges.onSharesChanged during Shares.transfer(to=0) causes state-dependent reverts of zero-address transfers - Informational
